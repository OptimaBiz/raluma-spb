# CLAUDE.md

## Проект
Raluma landing / preview.

Цель проекта: современный адаптивный лендинг для тестирования конверсии по услугам безрамного остекления.

Главный приоритет: коммерческая эффективность, понятный оффер, доверие, mobile UX, аккуратная адаптивная верстка.

## Роль Claude
- Работай как сильный frontend / UX / CRO инженер.
- Сначала понимай задачу и текущую реализацию, потом меняй код.
- Делай точечные, качественные изменения без хаотичного рефакторинга.
- Не «улучшай всё подряд», если задача локальная.
- Сохраняй текущую структуру проекта, если нет прямой причины менять.

## Workflow
- Перед изменениями проверь текущую ветку и статус репозитория.
- Сначала изучи связанные файлы: HTML, CSS, JS и существующие паттерны.
- Для сложных задач сначала дай краткий план.
- Новая задача: старт от `main`.
- Follow-up задача: продолжай ветку открытого PR.
- Один таск = одна ветка = один PR.
- Никогда не пушь напрямую в `main`.
- Не создавай новый PR для follow-up, если уже есть релевантный открытый PR.

## GitHub MCP
- GitHub MCP использовать только для проверки GitHub-состояния:
  - ветки
  - PR
  - коммиты
  - diff
  - файлы в репозитории
- Код менять через рабочее дерево VS Code / локального репозитория.
- Не использовать GitHub MCP как основной способ редактирования кода.
- Не пушить в `main`.

## Scope discipline
- Меняй только файлы и блоки, которые относятся к задаче.
- Не делай попутный редизайн соседних секций.
- Не меняй тексты, стили и JS-логику вне scope задачи.
- Не переименовывай классы/селекторы без необходимости.
- Не ломай уже работающую мобильную нижнюю панель, формы, CTA и аналитику.
- Не удаляй существующую функциональность без явного требования.

## UI / UX / CRO правила
- Mobile-first: сначала думай о мобильном сценарии.
- Desktop не должен ухудшаться из-за mobile-правок.
- Первый экран должен за 3 секунды объяснять:
  - что продается
  - для кого
  - в каком регионе
  - в чем выгода
  - что делать дальше
- CTA должны быть заметными, конкретными и не конфликтовать друг с другом.
- Primary CTA должен быть один главный в рамках смыслового блока.
- Не добавлять визуальный шум, лишние бейджи, карточки и декоративные элементы без пользы.
- Важны: доверие, премиальность, чистая сетка, читаемость, уверенный коммерческий тон.
- Не делать «дизайн ради дизайна».
- Любой блок должен помогать продаже или объяснению продукта.

## Mobile UX
- На мобильных не ломать sticky / bottom CTA panel.
- Не упоминать телефон внутри мобильных блоков, если телефон уже вынесен в отдельную нижнюю панель.
- Избегать длинных текстовых простыней.
- Проверять высоты, отступы, переносы, кликабельность и порядок CTA.
- Не допускать наложений, скачков верстки и двойных CTA в одном сценарии.
- Мобильная версия должна быть самостоятельной, а не просто сжатым desktop.

## Тексты
- Тексты должны быть продающими, конкретными и без воды.
- Заголовок объясняет ценность.
- Подзаголовок усиливает заголовок, а не повторяет его.
- CTA должен быть действием, а не абстракцией.
- Не использовать канцелярит, слабые обещания и общие фразы.
- Для Raluma важны формулировки про:
  - архитектуру дома
  - премиальный внешний вид
  - защиту от ветра и осадков
  - комфорт использования
  - аккуратный монтаж
  - подбор системы под объект
  - Санкт-Петербург и Ленинградскую область

## Верстка и код
- Следуй существующим паттернам HTML/CSS/JS.
- Не вводи новые библиотеки.
- Не переписывай весь `index.html`, если нужна точечная правка.
- CSS писать аккуратно, с понятной зоной ответственности.
- Не дублировать стили без необходимости.
- Не ломать существующие классы, если они используются JS.
- Если добавляешь класс — проверь, не конфликтует ли он с текущими стилями.
- JS менять только если задача требует изменения поведения.
- Не трогать формы, обработчики заявок и аналитику без прямого требования.

## Проверки после изменений
После правок обязательно проверить:

- `git diff`
- что изменены только нужные файлы
- desktop layout
- mobile layout
- CTA-сценарии
- формы и клики, если задача касается интерактива
- отсутствие console errors, если запускался браузер
- отсутствие случайных изменений вне scope

Если проект не имеет npm/build-команд, не выдумывать проверки. Честно указать, что проверено вручную.

## Финальный ответ после задачи
Всегда отвечай структурно:

1. Что изменено
2. Какие файлы изменены
3. Что именно сделано
4. Что не менялось сознательно
5. Какие проверки выполнены
6. Риски / замечания

## Запрещено
- Не пушить напрямую в `main`.
- Не делать массовый рефакторинг без задачи.
- Не менять unrelated sections.
- Не ломать mobile bottom panel.
- Не ухудшать читаемость ради визуальных эффектов.
- Не добавлять зависимости.
- Не заявлять, что тесты или проверки выполнены, если они не выполнялись.
- Не использовать GitHub MCP для редактирования кода вместо локального рабочего дерева.
